# 収益ゼロを「想定内」とした根拠

SEOアフィリエイトの収益化タイムラインは以下が通説。

```
0〜3ヶ月: インデックス蓄積期
3〜6ヶ月: 検索評価が上がり始める
6ヶ月〜: 収益発生の現実的ラインに到達
```

3ヶ月時点でPVが少ないのは設計の問題ではなく、SEOのタイムラグだ。

ただし「想定内」と言えるのは、**KPIを収益以外にも設定していたから**。

# 3つのKPI

## KPI 1: パイプライン稼働率

```
稼働率 = 正常稼働日数 / 総日数 × 100
```

3ヶ月（92日）のうちの停止日：

| 停止イベント | 停止時間 | 原因 | 対応 |
|:---|:---|:---|:---|
| cron 設定ミス | 6時間 | タイムゾーン設定誤り | UTC/JST変換を修正 |
| 記事生成エラー | 2時間 | Gemini APIレスポンス形式変更 | パーサー修正 |
| VPSメモリ不足 | 4時間 | ログファイルが肥大化 | logrotate設定 |
| X API レートリミット | 8時間 | 月間上限に達した | 投稿数を調整 |

**稼働率: 約97.8%**（92日中2日相当の停止）

目標は95%以上だったのでクリア。

## KPI 2: 記事承認率（≒プロンプト品質）

```
承認率 = 1回確認で承認できた記事数 / 総生成記事数 × 100
```

| 月 | 生成 | 承認 | 差し戻し | 承認率 |
|:---|:---|:---|:---|:---|
| 1ヶ月目 | 18本 | 11本 | 7本 | 61% |
| 2ヶ月目 | 20本 | 16本 | 4本 | 80% |
| 3ヶ月目 | 19本 | 17本 | 2本 | 89% |

承認率が61% → 89%に改善。差し戻しの主な原因は毎回記録して、プロンプトにフィードバックしている。

## KPI 3: 問題解決時間

```
平均解決時間 = 全問題の解決時間合計 / 問題発生件数
```

| 月 | 問題件数 | 平均解決時間 |
|:---|:---|:---|
| 1ヶ月目 | 8件 | 約4.5時間 |
| 2ヶ月目 | 6件 | 約2.8時間 |
| 3ヶ月目 | 4件 | 約1.9時間 |

問題自体の数が減り、解決速度も上がっている。問題発生時はすべて `docs/SKILL_AND_FAILURE_LOG.md` に記録して、同じエラーで2度詰まらないようにしている。

# 失敗ログ（抜粋）

## fail-001: 時刻ずれによる全記事が深夜に投稿された

**症状**: 朝6時投稿のはずが深夜2時に投稿された

**原因**: VPSのタイムゾーンはUTC。cronの時刻をJSTで設定していた

**修正**:
```bash
# crontab -e
# JST 06:00 = UTC 21:00（前日）
0 21 * * * cd /opt/cursor/cbd && python3 ...
```

## fail-002: Gemini APIのレスポンス形式変更でパーサーが壊れた

**症状**: 記事生成がすべてエラーになった

**原因**: APIのレスポンスの `candidates[0].content.parts[0].text` の構造が変わった

**修正**:
```python
# 堅牢なパーサーに変更
def parse_gemini_response(response) -> str:
    try:
        return response.candidates[0].content.parts[0].text
    except (AttributeError, IndexError, KeyError):
        return response.text  # フォールバック
```

## fail-003: ログファイルが肥大化してVPSディスクフル

**症状**: systemdサービスがクラッシュ・再起動を繰り返した

**原因**: Python の `logging` がデフォルトでローテーションしない

**修正**:
```python
from logging.handlers import RotatingFileHandler

handler = RotatingFileHandler(
    'app.log',
    maxBytes=10*1024*1024,  # 10MB
    backupCount=5
)
```
